building an opencode tui plugin

July 2026 • 8 min read

what is a tui plugin

opencode is an open-source AI coding agent that runs in the terminal, on the desktop, and inside your IDE. its terminal UI (TUI) is built on opentui — a framework for building rich terminal interfaces with solidjs. the TUI has a slot system: named places in the layout where plugins can inject their own UI. one of those slots is sidebar_content, which renders content in the left sidebar during an active coding session.

this post walks through building a real TUI plugin from scratch: a file tree sidebar. you will see the project structure, the module shape, the critical gotcha with solidjs jsx compilation, and the config traps that nearly derailed the whole thing. by the end you will know exactly what it takes to ship an opencode TUI plugin.

project structure

a TUI plugin is an npm package with a ./tui export. the loader resolves this export at runtime, imports the module, and calls the tui() factory function. here is the minimal file layout: 

opencode-file-tree/
  package.json
  build.mjs
  tsconfig.json
  src/
    index.tsx
  dist/
    index.js

the package.json is the most important piece. it must declare the ./tui export and the oc-plugin field so opencode knows this package targets the TUI system: 

{
  "name": "opencode-file-tree",
  "type": "module",
  "exports": {
    "./tui": {
      "default": "./dist/index.js"
    }
  },
  "oc-plugin": ["tui"],
  "dependencies": {
    "@opentui/solid": "^0.1.92",
    "solid-js": "^1.9.10"
  },
  "devDependencies": {
    "@opencode-ai/plugin": "^1.3.5"
  }
}

notice the dependencies. @opentui/solid is the solidjs renderer for opentui — it provides all the JSX elements like <box>, <text>, <scrollbox>, and the control flow components <Show> and <For>. solid-js provides the reactive primitives: createSignal, createMemo, createEffect. @opencode-ai/plugin provides the TypeScript types (TuiPlugin, TuiPluginApi, TuiPluginModule).

the build script uses bun build with the opentui solid plugin. this is the line that caused the most trouble: 

import solidPlugin from '@opentui/solid/bun-plugin'

const result = await Bun.build({
  entrypoints: ['./src/index.tsx'],
  outdir: './dist',
  target: 'bun',
  format: 'esm',
  external: [
    'solid-js',
    '@opentui/solid',
    '@opentui/core',
    '@opentui/keymap',
  ],
  plugins: [solidPlugin],
})

the external array tells bun not to bundle these packages — they are resolved from the host's runtime instead. the solidPlugin from @opentui/solid/bun-plugin is what makes the JSX transform work correctly. we will see what happens when you forget it.

the plugin anatomy

the entry point exports an object with id and tui. the id is required for path-based plugins and optional for npm plugins (the package name is used as fallback). the tui function receives three arguments: the api, plugin options, and meta metadata. 

/** @jsxImportSource @opentui/solid */
import type { TuiPlugin, TuiPluginModule } from '@opencode-ai/plugin/tui'
import { createMemo, createSignal, For, Show } from 'solid-js'
import { readdir, readFile } from 'node:fs/promises'
import { join, relative } from 'node:path'

const tui: TuiPlugin = async (api) => {
  const rootPath = api.state.path.worktree
    || api.state.path.directory
    || process.cwd()

  const [dirCache, setDirCache] = createSignal({})
  const [expanded, setExpanded] = createSignal(new Set())
  const [loadError, setLoadError] = createSignal()

  async function loadDir(dirPath) {
    const entries = await readDir(dirPath)
    setDirCache((prev) => ({ ...prev, [dirPath]: entries }))
  }

  loadDir(rootPath).catch((e) => setLoadError(e.message))

  const flatTree = createMemo(() => {
    /* walk cached directories, flatten with depth */
  })

  api.slots.register({
    slots: {
      sidebar_content() {
        return (
          <box gap={0}>
            <Show when={!loadError()} fallback={/* error */}>
              <Show when={flatTree().length > 0} fallback={/* loading */}>
                <For each={flatTree()}>
                  {(item) => /* file row */ }
                </For>
              </Show>
            </Show>
          </box>
        )
      },
    },
  })
}

export default { id: 'file-tree-sidebar', tui }

the /** @jsxImportSource @opentui/solid */ pragma tells bun to use opentui's JSX runtime. the sidebar_content function is a solidjs component. it reads signals created in the outer tui() scope and returns JSX using opentui elements. the slot system calls this function during rendering, passing { session_id } as props.

the jsx pitfall

this is the bug that took the longest to track down. the @opentui/solid/bun-plugin is not optional — it is required. without it, bun's default JSX transform compiles <Show> and <For> into React.createElement(Show, ...) calls. the opentui solidjs reconciler does not understand React.createElement. it expects _$createComponent(Show, ...) — the solidjs internal JSX runtime.

the symptom is silent: the plugin loads, the slot registers, but the sidebar renders nothing. no error message, no crash. the reconciler encounters React.createElement(Show, ...) and throws an Unknown component type error that is caught and swallowed by the slot system's error boundary. the sidebar stays empty.

here is the difference in compiled output: 

// without solidPlugin — broken
import { createElement as _$createElement } from "react";
_$createElement(Show, { when: loading, fallback: ... })

// with solidPlugin — correct
import { createComponent as _$createComponent } from "@opentui/solid";
_$createComponent(Show, { get when() { return !loadError() } })

notice two things: the import source changes from react to @opentui/solid, and the function changes from createElement to createComponent. the solid plugin also wraps prop accessors in getters (get when()) so that the reconciler can track reactive dependencies properly.

the fix is a single line in build.mjs: 

import solidPlugin from '@opentui/solid/bun-plugin'

// add to the Bun.build() call:
plugins: [solidPlugin]

this is easy to miss because the build succeeds either way. the JSX outputs valid JavaScript in both cases. only at runtime, inside the solidjs reconciler, does the difference surface.

the config gotchas

once the JSX is correct, the plugin still needs to be loaded by the TUI host. this is where two config traps await.

trap one: wrong config file. opencode has two separate config systems. opencode.json is for server/runtime plugins (tools, hooks, MCP servers). tui.json is for TUI plugins (sidebar slots, routes, keybinds). putting "opencode-file-tree" in opencode.json registers it as a server plugin — the ./tui export is never resolved, the tui() factory is never called, and the slot is never registered. the fix is trivial but easy to overlook: 

// — wrong
~/.config/opencode/opencode.json
{ "plugin": ["opencode-file-tree"] }

// — correct
~/.config/opencode/tui.json
{ "plugin": ["opencode-file-tree"] }

trap two: npm resolution. when the TUI host encounters a bare npm spec like "opencode-file-tree", it calls Npm.add("opencode-file-tree@latest"). this installs the package from the npm registry. if you have a fixed build node_modules locally but have not published it, the host downloads the old published version and overwrites your local files. the fix during development is to use a file path: 

// bypasses npm registry, loads local build
{ "plugin": ["./node_modules/opencode-file-tree/dist/index.js"] }

once the fixed version is published to npm, switch back to the bare spec. the rest of this post covers the publish step.

publishing

publishing is straightforward. bump the version in package.json, then run: 

npm publish --legacy-peer-deps

the --legacy-peer-deps flag avoids peer dependency conflicts with opentui's solid-js@1.9.12 requirement. after publishing, update tui.json back to the bare npm spec so the TUI host resolves the latest version from the registry.

you can also add engines.opencode to declare compatibility: 

"engines": {
  "opencode": "^1.0.0"
}

if the range is not satisfied, opencode skips the plugin with a warning. this is optional but recommended for production plugins.

what we learned

building an opencode TUI plugin is straightforward once you know the pieces:
package.json with exports["./tui"] and "oc-plugin": ["tui"]
build.mjs with solidPlugin from @opentui/solid/bun-plugin
entry point exporting { id, tui }
slot registration via api.slots.register()
config in tui.json, not opencode.json

the silent JSX bug is the most dangerous — it builds, loads, and registers without error, then produces an invisible sidebar. always verify the compiled output contains _$createComponent calls from @opentui/solid, not createElement from react.

Till then, keep experimenting :)
